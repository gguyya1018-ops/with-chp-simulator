1단계: 준비

준비 · 1회 실행

최적화를 시작하기 전에 필요한 모든 정보를 모으는 단계입니다.

건물의 1년치 열수요, 외부에서 받는 열, 전기 시장가격, 연료값을 365일분 준비합니다. 그리고 "축열조에 열이 얼마나 남아있고, CHP가 얼마나 쉬었는가"를 표현하는 상태공간을 정의합니다.

마지막으로 CHP를 각 부하(20%~100%)로 1시간 돌렸을 때 비용이 얼마나 드는지를 미리 계산해둡니다. 이렇게 해야 2단계에서 수십만 번의 비교를 빠르게 할 수 있습니다.

이 단계에서 결정되는 것

• 상태 개수: 축열조 = (최대용량 − 최소잔량) / 10 + 1 개 × 정지레벨 4단계
• 부하 옵션: 최소부하(20%) ~ 100%, 10% 단위 = 9개
• 시간 후보: 미가동(0h), 4h, 5h, ..., 24h = 21개

2단계: 역방향 탐색 (핵심)

핵심 알고리즘 · 매일 반복 (MPC)

이 단계가 정밀 최적화의 핵심입니다. MPC 방식에서는 매일 오늘부터 30일 앞까지를 역방향으로 탐색합니다.

왜 거꾸로 갈까요? 30일째의 최선의 선택은 그 날 하루만 보면 됩니다. 29일째는 "오늘 + 내일(이미 답을 알고 있음)"을 보면 됩니다. 이렇게 거슬러 올라가면 오늘에 도달했을 때 30일 전체를 고려한 최적 결정이 완성됩니다.

그리고 오늘의 결정만 확정하고, 내일은 실제 축열조 잔량으로 다시 30일 앞을 계산합니다. 이것이 MPC입니다.

이 방식을 동적 프로그래밍(Dynamic Programming)이라고 합니다. "큰 문제를 작은 문제로 나누어 풀고, 이미 푼 결과를 재활용한다"는 수학적 기법입니다.

3단계: 순방향 실행

실행 · 365일 반복

2단계에서 오늘의 최적 결정이 나왔습니다. 이제 그 결정을 실제로 적용합니다.

매일 다음을 수행합니다:

① DP에서 나온 최적 결정(가동시간, 부하) 적용

365일 데이터 로드

처리

시뮬레이션에 필요한 1년치 데이터를 모두 모읍니다. 각 날에 대해 다음 정보가 준비됩니다:

• 시간별 열수요 — 건물이 매 시간 소비하는 열량 (24시간 패턴)
• 외부수열량 — 소각장, ERG 등에서 무상/유상으로 받는 열
• 시간별 전력가격 — CHP가 생산한 전기를 한전에 팔 때의 가격
• NG(천연가스) 단가 — CHP와 PLB의 연료비
• 공휴일/주말/정비일 구분

app.js buildDailyData

상태공간 정의 (어떤 상황들을 비교할지)

처리

최적화가 비교해야 할 "현재 상황"이 몇 가지나 있는지를 정하는 단계입니다.

같은 1월 15일이라도 "축열조에 열이 많이 남아있는 상황"과 "거의 바닥인 상황"은 전혀 다른 결정이 필요합니다. 또한 "어제 CHP를 돌린 상황"과 "3일째 쉬고 있는 상황"도 다릅니다 (재기동 비용이 다르니까). 이런 모든 경우를 빠짐없이 비교하기 위해 범위를 나누는 겁니다.

축열조는 최소잔량~최대용량 범위를 10Gcal 단위로 나누어 구간으로 관리합니다.

정지레벨은 CHP가 쉰 기간에 따라 4단계로 나뉩니다:

ON — 현재 가동 중
Hot — 1일 이내 정지 (재기동 비용 낮음)
Warm — 2일 이내 정지 (재기동 비용 중간)
Cold — 3일 이상 정지 (재기동 비용 높음)

탐색 규모

① 상태 수
축열조 구간 수 × 정지레벨 4단계
예) 최대 850, 최소 100 → (850−100)/10+1 = 76구간
→ 76 × 4 = 304가지 상태

② 조합 수
부하 9가지 × 가동시간 21가지 = 189가지 조합

③ 탐색량
하루 = 304 × 189 = 약 5.7만 번
연간 = 5.7만 × 365일 = 약 2,100만 번

숫자가 커 보이지만 단순한 더하기/빼기 연산이라 컴퓨터는 1~2초면 끝냅니다.

일별 비용 사전계산

비교

CHP를 돌리면 가스를 태워서 돈이 나가고(연료비), 전기가 만들어져서 돈이 들어옵니다(전기 판매). 이 둘의 차이가 "CHP를 1시간 돌리는 실제 비용"입니다.

이 값은 날마다 다릅니다. 가스값은 월마다 바뀌고, 전기가격은 매일 달라지며, CHP 부하에 따라서도 가스 소모량과 발전량이 다릅니다.

계산 공식

시간당 순비용 = 가스비/시간 − 전기매출/시간

음수 → CHP를 돌릴수록 이익 (전기 수익 > 연료비)
양수 → CHP를 돌리면 비용 발생

PLB(보조보일러)의 열 1Gcal당 생산 단가도 함께 계산합니다.

시간별 사전검증

검증

DP가 하루 총량만 보고 "24시간 가동 OK"라고 판단했지만, 실제로는 새벽처럼 수요가 낮은 시간에 축열조가 넘치는 문제가 있었습니다.

이를 해결하기 위해, DP가 결정하기 전에 실제 시간별 시뮬레이션을 미리 돌려서 넘치지 않는 최대 CHP 가동시간을 확인합니다.

축열조 800Gcal에서 CHP 80% 부하:

24h → 새벽 2시에 넘침 → 탈락
23h → 새벽 1시에 넘침 → 탈락
...
18h → 넘침 없음 → 최대 18h 허용
절대 규칙: 이 검증을 통해 시간별 넘침을 원천 차단합니다
app.js calcMaxSafeHours

모든 조합 시도

비교

각 상태(축열조 잔량 × 정지레벨)에서 가능한 모든 "부하 × 가동시간" 조합을 시도합니다.

각 조합마다: ① CHP 비용 ② 기동비용 ③ 축열조 변화 ④ PLB 필요 여부 ⑤ 내일 이후 미래 비용을 합산하여 총비용이 가장 작은 조합을 선택합니다.

CHP 비용 계산

액션
열생산량 = 가동시간 × 시간당 열출력
비용 = 가동시간 × 시간당 순비용

순비용이 음수이면 CHP를 오래 돌릴수록 이익입니다.

기동비용 판단

판단

CHP가 쉬고 있다가 다시 켜면 보일러를 데우는 데 가스가 소모됩니다. 쉰 기간이 길수록 더 많이 듭니다.

Hot (1일 이내 정지): 4시간 × 300 Nm³/시간 × NG단가
Warm (2일 이내 정지): 6시간 × 350 Nm³/시간 × NG단가
Cold (3일 이상 정지): 8시간 × 400 Nm³/시간 × NG단가
app.js calcStartupCost

축열조 변화 계산

액션
새 잔량 = 현재 잔량 + CHP 열생산 − 하루 열수요

이 값에 따라 내일의 시작 상태가 결정됩니다.

축열조 부족 → PLB 투입

판단

CHP만으로 축열조가 최소 수준 아래로 떨어지면 PLB를 강제 투입합니다.

절대 규칙: 축열조 잔량은 어떤 시간에도 최소 수준 아래로 내려가면 안 됩니다

축열조 넘침 → 조합 제외

판단

축열조가 상한을 초과하면 그 조합은 아예 후보에서 탈락시킵니다.

절대 규칙: 축열조 잔량은 상한을 초과할 수 없습니다 — 열 버림 금지

최소 비용 결정 기록

액션
총비용 = CHP 비용 + 기동비 + PLB 비용 + 미래 최소비용

→ 이 값이 가장 작은 (부하, 가동시간) 조합을 기록

"미래 최소비용"은 다음날의 탐색 결과(이미 계산 완료)입니다.

최적 결정 추출

처리

현재 실제 축열조 잔량과 정지 상태를 기반으로, DP 결과 테이블에서 오늘의 최적 결정을 꺼냅니다.

CHP 블록 배치 (Stage 2)

처리

"오늘 15시간 가동"이라고 결정되었다면, 하루 24시간 중 어느 시간대에 CHP를 돌릴지를 정합니다.

모든 가능한 시작시간을 시뮬레이션하고, 전기 시장가격이 높은 시간대에 배치합니다. 축열조가 넘치는 배치는 무조건 거부합니다.

절대 규칙: 축열조가 넘치는 배치는 완전 거부

PLB 투입 판단 (Stage 3)

판단

CHP 배치 후, CHP + 외부수열 + 축열조만으로 하루를 버틸 수 있는지 24시간 시뮬레이션으로 검증합니다. 부족하면 PLB를 투입합니다.

PLB는 부족분의 2배를 초과하지 않는 범위에서, 부족 시점 직전에 배치합니다.

24시간 시뮬레이션 (Stage 4)

처리

모든 결정이 끝나면 24시간을 1시간씩 실제로 시뮬레이션합니다. 매 시간 축열조에 들어오는 열과 나가는 열을 계산합니다.

시뮬 도중 축열조가 상한에 도달하면 CHP 열출력을 실시간으로 줄여서 넘치지 않게 합니다.

일별 집계

액션

• CHP 연료비 — 가동시간 × 시간당 가스비
• 전력판매 — 시간별 (전기가격 × 발전량) 합산
• 기동비용 — 정지 후 재기동 시 발생
• PLB 연료비 — PLB 가동 시 가스비
• CHP단가 — (연료비 + 기동비 − 전력판매) / 생산열량

축열조 최종 잔량은 내일 MPC의 시작값이 됩니다.

각 단계 설명

1단계: 데이터 입력

SMP(시간별), 판매실적, 외부거래, 용수약품, 공휴일, 전력수급 데이터와 단가/요금 테이블(CP, NG단가, 전기요금, 열요금, 수도/약품단가 등)을 Excel로 일괄 업로드하거나 개별 입력합니다. 필수 데이터: SMP, 외부거래, NG단가, CP/TLF.

2단계: 설정

CHP 성능표(부하별 열/전기/가스), PLB 성능표, 축열조(용량/초기잔량/최소유지), 기동조건(Hot/Warm/Cold), 정비일정, 외부수열원 레지스트리, 단가산정 계수, 전력수요 패턴(월별 평균kW + 24시간 패턴) 등 8개 스텝으로 설정합니다.

3단계: 데이터 산출

입력 데이터와 설정을 결합하여 중간 계산을 수행합니다. MP(정산가격) = SMP x TLF + 무부하비정산, 일별 용량요금(CP), 외부수열비용, 열판매수입, 용수약품비가 산출됩니다.

함수: runAllCalc → calculateMP, calculateCapacityCharge, calculateExternalHeatCost, calculateHeatSalesRevenue, calculateWaterChem

4단계: 기동계획 (최적화)

365일 각각에 대해 CHP 가동시간/부하율을 결정하는 핵심 단계입니다. MPC 방식으로 매일 30일 앞을 DP로 탐색한 후 오늘의 결정만 확정합니다.

함수: solveMPC (동기), solveMPCAsync (비동기) → "최적화 흐름" 탭에서 상세 확인

5단계: 비용산출

최적화 결과를 바탕으로 전력비(기본료/수전요금/기후환경/연료비/전력기금), 가스비, 용수약품비, 수열비를 월별/연간으로 집계합니다.

함수: renderAllCosts

6단계: 운영손익

수입(전력판매 + 용량요금 + 열판매)에서 비용(연료비 + 기동비 + 수열비 등)을 차감하여 공헌이익을 산출합니다. 월별 차트와 일별 상세 보기를 제공합니다.

함수: renderContribution

MPC 작동 원리

MPC란?

Model Predictive Control (모델예측제어). 내비게이션에 비유하면: "목적지까지 30km 앞을 보고 지금 어느 길로 갈지 결정하되, 1km 가면 다시 30km 앞을 보고 다시 판단하는 것"입니다.

위드시뮬에서는 오늘부터 30일 앞까지를 DP(동적 계획법)로 분석하여 가장 비용이 적은 경로를 찾되, 오늘 결정만 확정합니다. 내일은 실제 축열조 잔량을 기준으로 다시 30일 앞을 탐색합니다.

제어공학/최적화 분야에서 널리 쓰이는 기법입니다. 발전소 급전계획, 건물 에너지 관리, 자율주행 등에 적용됩니다.

왜 MPC를 쓰나?

방식설명문제점
근시안(Greedy)오늘만 보고 결정내일 수요 폭증 대비 불가
전체 DP365일 한번에 계산실제 축열조 상태 미반영
MPC30일 앞을 보되 오늘만 실행둘의 장점 결합

매일 반복하는 7단계

MPC 루프 안에서 하루마다 아래 7단계가 순서대로 실행됩니다.

1
데이터 준비
365일 전체의 일별 데이터를 미리 구성합니다. 수요(시간별), 외부수열, SMP/MP, NG단가, 공휴일, 정비일 등을 하나의 배열로 통합합니다.
입력: DATA 객체, 설정값
출력: dayInfo 배열 (365개)
관련: buildDailyData
2
사전검증 (overflow 방지)
현재 축열조 잔량에서 CHP를 가동했을 때 넘치지 않는 최대 안전 가동시간을 계산합니다. 이 값이 DP 탐색의 상한이 됩니다.
입력: 축열조 잔량, 수요 패턴, CHP 열출력
출력: 날/잔량별 최대 안전 시간 (0~24)
관련: calcMaxSafeHours
3
역방향 DP (30일 윈도우)
오늘부터 30일(horizon) 범위에서, 마지막 날부터 오늘로 거꾸로 계산합니다. 각 날/축열조 잔량/정지 상태 조합마다 최소비용 경로를 구합니다.
입력: dayInfo, 사전검증 결과, DP 상태공간
출력: 오늘의 최적 결정 테이블
관련: DP 역방향 루프 (Bellman Equation)
4
결정 추출
현재 실제 축열조 잔량과 정지 상태 기반으로, DP 결과에서 오늘의 최적 결정을 꺼냅니다. "CHP 18시간, 80% 부하" 등.
입력: 실제 축열조 잔량, 정지레벨
출력: CHP 가동시간, 부하율
5
블록 배치 (Stage 2)
결정된 가동시간을 하루 24시간 중 어디에 배치할지 정합니다. MP가 높은 시간대에 가동하여 수익을 극대화합니다.
입력: CHP 시간/부하, 시간별 MP
출력: CHP 시작시각
6
PLB 판단 (Stage 3)
CHP 배치 결과로 축열조를 시뮬레이션하여, 최소 잔량 아래로 떨어지면 보조보일러를 투입합니다.
입력: CHP 블록 배치, 시간별 수요
출력: PLB 대수, 시간, 시작시각
7
시간별 시뮬 (Stage 4)
0시~23시까지 1시간씩 실제 운전을 시뮬레이션합니다. 이 날의 마지막 축열조 잔량이 내일 MPC의 시작값이 됩니다.
입력: CHP/PLB 블록, 시간별 수요
출력: 24시간 상세 기록, 일별 집계
이 7단계가 365일 동안 반복됩니다. 전체 소요시간은 약 3~10초이며, UI에서는 워크플로우 오버레이로 진행 상태를 시각화합니다.

데이터 흐름도

시뮬레이터에 입력되는 데이터는 크게 입력 데이터(실적/시장)와 단가/요금 데이터로 나뉩니다. 모든 데이터는 전역 DATA 객체에 저장됩니다.

사용: 열판매수입, 용수약품비 계산

핵심 과제: 30년치 데이터 확보

실적 데이터는 과거 1~2년뿐입니다. 미래 데이터를 추정해야 합니다.

추정 방법
  • 수요: 기준년도 패턴 반복 + 연도별 성장률
  • SMP: 기준년도 패턴 + 연도별 물가상승 반영
  • NG단가: 기준년도 + 연도별 상승률 (에너지 가격 시나리오)
  • 외부수열: 기준년도 패턴 반복 + 신규 수열원 반영 시점
  • 기기 성능: 경년 열화 모델 (효율 감소, 정비 주기 변경)

단계별 확장 방안

1단계: 데이터 생성기 구현
기준년도 실적을 복제하여 N년치 데이터를 자동 생성하는 함수 구현. 연도별 성장률/물가상승 파라미터를 설정 UI에 추가. buildDailyData의 연도 루프를 다년도로 확장.
2단계: 성능 최적화
MPC 루프를 Web Worker로 분리하여 UI 블로킹 방지. DP 내부 Float64Array 최적화는 이미 완료. 진행률 표시를 10,950일 기준으로 조정.
3단계: 결과 집계/시각화 확장
월별 요약을 연별 요약으로 확장. 30년 추세 차트(연간 비용, 공헌이익 추이) 추가. NPV(순현재가치), IRR(내부수익률) 등 투자분석 지표 산출.
4단계: 시나리오 비교
복수 시나리오(낙관/기본/비관)를 설정하여 30년 결과를 비교. 기기 교체 시점, 수열원 변경 등의 이벤트를 타임라인에 설정.
MPC의 장점: horizon 30일을 유지하면 DP 상태공간은 변하지 않습니다. 30년이라도 매일 30일짜리 소형 DP를 10,950번 풀 뿐이므로 확장에 유리합니다.

용어 사전

MPC (Model Predictive Control)
모델예측제어. 매 시점 미래 N일을 예측하여 최적 결정을 내리되, 한 단계만 실행하고 다음 시점에 다시 예측하는 제어 방식.
DP (Dynamic Programming)
동적 계획법. 큰 문제를 작은 부분 문제로 나누어 최적해를 구하는 알고리즘. 역방향으로 각 상태의 최소비용을 계산.
SMP (System Marginal Price)
계통한계가격. 전력시장에서 시간별로 결정되는 전기 도매가격. CHP 전기 판매 수익 기준.
MP (Market Price / 정산가격)
실제 정산가격. SMP x TLF + 무부하비정산으로 산출.
TLF (Transmission Loss Factor)
송전손실보전계수. 발전소에서 소비지까지 전력 손실 보상 계수. 평일/토요일/공휴일별로 다름.
CP (Capacity Payment)
용량요금. 발전 용량 확보에 대한 고정 보상금. 가동 여부와 무관하게 받는 수입.
CHP (Combined Heat and Power)
열병합발전. 가스를 태워 열과 전기를 동시에 생산. 전기는 판매, 열은 지역난방에 공급.
PLB (Peak Load Boiler)
첨두부하보일러. 열수요가 CHP만으로 부족할 때 보조적으로 가동. 전기 미생산.
축열조 (Thermal Storage)
열을 온수 형태로 저장하는 탱크. 생산과 소비의 시간차를 완충. 최소잔량/최대용량 제약.
Horizon (탐색 범위)
MPC에서 미래를 내다보는 일수. 현재 30일. 이 범위 내에서 DP로 최적 경로 탐색.
netRequired (순필요열량)
수요 - 외부수열 - 인테코 순수열. CHP/PLB/축열조가 충당해야 할 열량.
stopLevel (정지레벨)
CHP 정지 경과 상태. 0=가동중, 1=Hot(1일), 2=Warm(2일), 3=Cold(3일+).
Bellman Equation
DP 핵심 점화식. "현재 비용 + 미래 최적 비용 = 전체 최적 비용". 역방향으로 채움.
Overflow / Underflow
축열조 상한/하한 이탈. overflow 시 열 버림, underflow 시 공급 부족.
